⚠️ EDU 4

Apply Lag exceeded

Apply was paused on stby01, 50,000 rows of redo were generated on prim01 (apply lag grew to ~5 min), and a failover was forced.

Lesson: FastStartFailoverLagLimit=0 protects Transport Lag, NOT Apply Lag. The failover succeeded despite the large Apply Lag — Oracle-correct, counter-intuitive.
⚠️ PART 5

Master Observer outage

Forced systemd stop of dgmgrl-observer-ext.service. We checked whether the 3-Observer quorum auto-promotes a backup.

Lesson: Explicit SET MASTEROBSERVER TO obs_dc worked OK. Auto-promotion was blocked by RestartSec=10s in systemd — the observer came back faster than the broker could detect the loss.

Caught only in real testing

5 bugs that surfaced only under real operation

These gaps would not appear on a single run — they only emerged with multiple role swaps and real load.

S28-64

APPPDB OPEN MODE persistence after switchover

Problem

SAVE STATE stores the PDB state per-role at a point in time. After a switchover the old state persists and APPPDB stays in MOUNTED.

Symptom

ORA-65054 when connecting to APPPDB after stby01 takes over the primary role.

Fix

srvctl modify pdb -policy AUTOMATIC -role PRIMARY on both clusters — CRS-managed auto-OPEN after a role swap.

S28-65

SCAN listener BLOCKED service after multiple role swaps

Problem

After 3+ role swaps the MYAPP_TAC service is BLOCKED on the SCAN listener. JDBC sees "ORA-12514 listener does not currently know of service".

Symptom

TestHarness_TAC connected to stby01 instead of prim01 (the service was registered on both). Routing did not respect -role PRIMARY.

Fix

srvctl stop service ... ; srvctl start service ... for clean re-registration. Fix targeted at Oracle Restart on SI.

S28-66

DGMGRL syntax change in 26ai for Master Observer promotion

Problem

Legacy 19c EDIT CONFIGURATION SET MASTEROBSERVER='<name>' does not work in 26ai — syntax error.

Symptom

"DGM-XXXX: invalid syntax" when manually promoting a backup observer in scenario 5.

Fix

26ai uses a new top-level syntax: SET MASTEROBSERVER TO <name>;. Deployment scripts updated.

S28-67

FastStartFailoverLagLimit protects Transport, NOT Apply

Problem

Assumption: "LagLimit=0 blocks failover whenever the standby has any lag." The assumption is WRONG.

Symptom

In scenario 4 apply was stopped, apply lag grew to 5 min — failover still succeeded because Transport Lag was 0 (redo had reached the SRL).

Fix

Educational lesson (no physical fix). MaxAvailability + AFFIRM = "redo arrived" ≠ "redo applied". To gate on Apply Lag, manual STARTUP MOUNT after failover (broker auto-reinstate).

S28-68

systemd RestartSec=10s blocks observer-outage tests

Problem

In scenario 5 we wanted to demonstrate auto-promotion of a backup observer. kill -9 on the Master did not help — systemd restarted the process before the broker noticed the loss.

Symptom

Auto-promotion did not occur because Master Observer came back faster than FastStartFailoverThreshold=30s.

Fix

Workaround: explicit SET MASTEROBSERVER TO obs_dc for the demo. In production: consider a larger RestartSec on the dgmgrl-observer-ext unit, but only if the outage demo is a priority.

Lessons learned

Lessons from autonomous MAA testing

3 perspectives on the same session: DBA, AI agent, DevOps/SRE.

🛠️

Value for the DBA

  • ~6–8× faster test execution than manual
  • Every bug = automatic lookup in EXECUTION_LOG + FIXES_LOG + auto-fix
  • Every iteration = a transparent trail in AUTONOMOUS_ACCESS_LOG.md
  • Operator approves strategic decisions, AI executes step by step
  • Fixes return to the script immediately — the next iteration is idempotent
🤖

What the AI agent learned

  • 26ai broker terminology has changed compared to 19c (S28-66)
  • Multi-Observer redundancy works flawlessly in operation
  • TAC LEVEL1 + Transaction Guard = 100% transparent replay
  • Auto-Reinstate in 26ai is reliable — even after multiple swaps
  • systemd RestartSec=10s can mask failures during tests
🚀

DevOps / SRE

  • The operator + AI agent + SSH equivalency pattern scales to production
  • New-DBA onboarding: read the live log instead of Confluence
  • Incident response: AI carries the context of past fixes + current environment state
  • Knowledge transfer: every session generates a permanent artifact
  • A typical stack for banking/telco/healthcare — 3-site MAA + TAC
